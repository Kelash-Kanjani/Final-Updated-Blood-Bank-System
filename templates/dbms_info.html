📐 Database Normalization

The database follows all three normal forms to eliminate redundancy and ensure data integrity.

1NF — First Normal Form
✅ Every table has a primary key. Every column holds atomic (single) values. No repeating groups. Blood group is a single value per donor, not a list.
TablePrimary KeyAtomic Columns
Donorid (AUTOINCREMENT)name, cnic, phone, email, blood_group — all single values
Stockid (AUTOINCREMENT)One row per donation batch — no multi-valued columns
Donationid (AUTOINCREMENT)One row per donation event — no repeating groups
Requestsid (AUTOINCREMENT)One row per hospital request
AuditLogid (AUTOINCREMENT)One row per system event
2NF — Second Normal Form
✅ No partial dependencies. Every non-key column depends on the whole primary key, not just part of it. Since all primary keys are single columns (no composite PKs), 2NF is automatically satisfied.
TableNon-Key ColumnsDepends On
Donationdonation_date, unitsFully on Donation.id ✅
Stockblood_group, units, datesFully on Stock.id ✅
Donorname, cnic, blood_groupFully on Donor.id ✅
3NF — Third Normal Form
✅ No transitive dependencies. No non-key column depends on another non-key column. Donor details (name, cnic) are not repeated in the Donation table — only donor_id is stored as a Foreign Key.
Design DecisionWhy it satisfies 3NF
Donation stores donor_id, not donor name or blood group Donor name depends on Donor.id, not on Donation.id — no transitive dependency
Stock stores blood_group directly (not a FK to Donor) Once blood is in stock it is independent of who donated it — correct separation
Hospital name stored in Requests as text (not a separate table) Hospital info is not further decomposed because no hospital attributes are tracked beyond name
Entity Relationship Summary
Donor (1) ──────── (M) Donation    [One donor can donate many times]
Donation (1) ───── (1) Stock batch   [Each donation creates one stock batch]
Stock (M) ─────── (1) Requests     [Multiple batches can fulfill one request]

🔍 Indexes

Indexes speed up SELECT queries. Without an index, SQLite scans every row (full table scan). With an index, it jumps directly to matching rows like a book index.

📦 Stored Procedures

SQLite does not have native stored procedure syntax, but the same concept is implemented as encapsulated Python functions. Each function wraps a complete multi-step database operation in a single transaction.

SP 1: sp_register_donor()

Purpose: Registers a new donor in the Donor table and returns the new donor_id.

Called by: Donor registration form (New Donor flow)

-- Equivalent SQL Stored Procedure:
CREATE PROCEDURE sp_register_donor(
    p_name TEXT, p_nationality TEXT, p_cnic TEXT,
    p_phone TEXT, p_email TEXT, p_blood_group TEXT, p_donor_code TEXT
)
BEGIN
    INSERT INTO Donor(donor_code, name, nationality, cnic, phone, email, blood_group)
    VALUES(p_donor_code, p_name, p_nationality, p_cnic, p_phone, p_email, p_blood_group);

    SELECT last_insert_rowid() AS new_donor_id;
END;
SP 2: sp_record_donation()

Purpose: Records a donation in both the Donation table and the Stock table atomically. If one insert fails, both are rolled back.

Called by: Donor form (both New and Existing donor)

-- Equivalent SQL Stored Procedure:
CREATE PROCEDURE sp_record_donation(
    p_donor_id INT, p_blood_group TEXT,
    p_units INT, p_donation_date TEXT, p_expiry_date TEXT
)
BEGIN
    BEGIN TRANSACTION;
        INSERT INTO Donation(donor_id, donation_date, units)
        VALUES(p_donor_id, p_donation_date, p_units);

        INSERT INTO Stock(blood_group, units, donation_date, expiry_date)
        VALUES(p_blood_group, p_units, p_donation_date, p_expiry_date);
    COMMIT;
EXCEPTION
    ROLLBACK;
END;
SP 3: sp_process_blood_request()

Purpose: Checks available stock, deducts using FIFO (oldest batches first), and records the request as Approved or Rejected — all in one transaction.

Called by: Blood Request form

-- Equivalent SQL Stored Procedure:
CREATE PROCEDURE sp_process_blood_request(
    p_hospital TEXT, p_blood_group TEXT, p_units INT
)
BEGIN
    DECLARE v_available INT;
    DECLARE v_status TEXT;

    SELECT SUM(units) INTO v_available
    FROM Stock WHERE blood_group = p_blood_group;

    IF v_available >= p_units THEN
        -- FIFO deduction from oldest batches
        -- (loop through batches by donation_date ASC)
        SET v_status = 'Approved';
    ELSE
        SET v_status = 'Rejected';
    END IF;

    INSERT INTO Requests(hospital, blood_group, units, status)
    VALUES(p_hospital, p_blood_group, p_units, v_status);

    SELECT v_status AS result;
END;
SP 4: sp_purge_expired_stock()

Purpose: Removes all stock batches past their expiry date. The trigger trg_after_stock_delete fires automatically for each deleted row.

Called by: Every page load (auto-maintenance)

-- Equivalent SQL Stored Procedure:
CREATE PROCEDURE sp_purge_expired_stock()
BEGIN
    DELETE FROM Stock
    WHERE expiry_date < date('now');
    -- trg_after_stock_delete fires automatically for each row
END;
